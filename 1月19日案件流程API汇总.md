## API概述

### 功能说明

案件流程API用于管理破产案件流程的7个阶段（立案阶段、接管阶段、调查阶段、债权阶段、重整阶段、分配阶段、终结阶段）的所有模块数据。系统采用统一的数据表存储所有阶段和模块的信息，通过阶段编号和模块编码进行区分。

### 使用场景

- 案件各阶段数据的录入和管理
- 案件流程进度的跟踪和查询
- 各阶段模块信息的维护
- 附件和特有字段数据的存储

### 技术特点

- 统一的API接口设计，简化前端开发
- 支持JSON格式存储附件和特有字段数据
- 软删除机制，保证数据安全
- 完整的参数校验和错误处理

## 1. 新增阶段数据

### 接口信息

- **请求路径**: `/`
- **请求方法**: `POST`
- **接口描述**: 新增案件流程阶段数据

### 请求参数

#### 请求体参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| caseId | Long | 是 | 案件ID |
| stageNum | Integer | 是 | 阶段编号(1-7) |
| stageName | String | 是 | 阶段名称 |
| moduleCode | String | 是 | 模块编码 |
| moduleName | String | 是 | 模块名称 |
| title | String | 否 | 标题 |
| content | String | 否 | 内容 |
| processDate | LocalDateTime | 否 | 处理日期 |
| attachments | String | 否 | 附件列表(JSON格式) |
| fieldData | String | 是 | 模块特有字段数据(JSON格式) |
| status | String | 否 | 状态(默认ACTIVE) |

#### 阶段编号说明

| 值 | 阶段名称 |
|----|----------|
| 1 | 立案阶段 |
| 2 | 接管阶段 |
| 3 | 调查阶段 |
| 4 | 债权阶段 |
| 5 | 重整阶段 |
| 6 | 分配阶段 |
| 7 | 终结阶段 |

#### 状态枚举值

| 值 | 说明 |
|----|------|
| ACTIVE | 激活 |
| INACTIVE | 停用 |
| DELETED | 删除 |

#### 请求示例

```http
POST /api/case-process-stage
Content-Type: application/json

{
  "caseId": 1,
  "stageNum": 2,
  "stageName": "接管阶段",
  "moduleCode": "STAGE2_PROPERTY_RECEIPT",
  "moduleName": "财产接收",
  "title": "财产接收记录",
  "content": "接收XX公司财产",
  "processDate": "2023-02-15T14:30:00",
  "attachments": "[{\"fileName\":\"财产清单.xlsx\",\"filePath\":\"/uploads/20230215/财产清单.xlsx\",\"fileType\":\"xlsx\",\"fileSize\":2048000}]",
  "fieldData": "{\"receiver\":\"李四\",\"receiveDate\":\"2023-02-15\",\"propertyType\":\"固定资产\",\"propertyValue\":10000000}",
  "status": "ACTIVE"
}
```

### 成功响应示例

```json
{
  "code": 200,
  "message": "success",
  "data": true
}
```

### 错误响应示例

```json
{
  "code": 400,
  "message": "案件ID不能为空",
  "data": null
}
```

## 错误码说明

### HTTP状态码

| 状态码 | 说明 |
|--------|------|
| 200 | 请求成功 |
| 400 | 请求参数错误 |
| 401 | 未授权 |

| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

### 业务错误码

| 错误码 | 说明 | 解决方案 |
|--------|------|----------|
| 200 | 操作成功 | - |
| 400 | 请求参数错误 | 检查请求参数格式和必填项 |
| 401 | 未授权 | 检查token是否有效 |
| 403 | 禁止访问 | 检查用户权限 |
| 404 | 资源不存在 | 检查请求的资源ID是否正确 |
| 500 | 服务器内部错误 | 联系后端开发人员 |

### 常见错误信息

| 错误信息 | 说明 | 解决方案 |
|----------|------|----------|
| 案件ID不能为空 | caseId参数为必填项 | 确保请求体中包含caseId字段 |
| 阶段编号不能为空 | stageNum参数为必填项 | 确保请求体中包含stageNum字段 |

| 模块特有字段数据不能为空 | fieldData参数为必填项 | 确保请求体中包含fieldData字段 |
| 阶段数据不存在 | 查询的阶段数据ID不存在 | 检查阶段数据ID是否正确 |

## 注意事项

### 接口调用频率限制

- 当前版本暂无接口调用频率限制
- 建议前端在调用接口时添加适当的防抖和节流机制
- 避免短时间内重复调用同一接口

### 权限要求

- 所有接口都需要有效的JWT token
- token通过请求头中的Authorization字段传递，格式为：Bearer {token}
- token过期后需要重新登录获取新token

### 数据格式约束

#### JSON字段格式

**attachments字段格式**：
```json
[
  {
    "fileName": "文件名.pdf",
    "filePath": "/uploads/20230110/文件名.pdf",
    "fileType": "pdf",
    "fileSize": 1024000
  }
]
```

**fieldData字段格式**：
fieldData字段为JSON字符串，存储模块特有字段数据，不同模块的字段结构不同。

示例（财产接收模块）：
```json
{
  "receiver": "李四",
  "receiveDate": "2023-02-15",
  "propertyType": "固定资产",
  "propertyValue": 10000000
}
```

#### 日期时间格式

- 日期时间字段使用ISO 8601格式：`yyyy-MM-ddTHH:mm:ss`
- 示例：`2023-02-15T14:30:00`

#### 字符串编码

- 所有字符串字段使用UTF-8编码
- 特殊字符需要进行URL编码

### 数据一致性

- 同一案件的不同阶段数据之间需要保持一致性
- 删除案件时，相关的阶段数据会级联删除
- 更新案件信息时，建议同步更新相关的阶段数据

### 性能优化建议

- 查询案件的所有阶段数据时，建议使用分页查询
- 对于大量数据的查询，建议添加适当的筛选条件
- 前端应实现数据缓存机制，减少不必要的接口调用

### 安全建议

- 敏感数据不要存储在fieldData字段中
- 附件文件路径不要包含敏感信息
- 建议在前端对用户输入进行数据验证

#### 模块编码规范

系统支持7个阶段共32个模块，模块编码格式为 `STAGE{阶段号}_{模块名英文大写}`：

**阶段一：立案阶段**
- STAGE1_REGISTER_INFO - 立案信息
- STAGE1_CASE_ACCEPT - 案件受理
- STAGE1_CASE_ASSIGN - 案件分配

**阶段二：接管阶段**
- STAGE2_PROPERTY_RECEIPT - 财产接收
- STAGE2_EMERGENCY_MANAGEMENT - 应急管理
- STAGE2_PROPERTY_PLAN - 财产方案管理
- STAGE2_PERSONNEL_EMPLOYMENT - 人员聘用
- STAGE2_INTERNAL_AFFAIRS - 内部事务管理
- STAGE2_CONTRACT_MANAGEMENT - 合同管理
- STAGE2_BUSINESS_MANAGEMENT - 营业管理

**阶段三：调查阶段**
- STAGE3_PROPERTY_INVESTIGATION - 财产调查
- STAGE3_SOCIAL_SECURITY - 社保费用
- STAGE3_TAX_VERIFICATION - 税收核定

**阶段四：债权阶段**
- STAGE4_MEETING_DOCUMENTS - 债权人会议文件
- STAGE4_CLAIM_CONFIRMATION - 债权确认
- STAGE4_REMUNERATION_PLAN - 报酬方案
- STAGE4_IMPORTANT_ACTIONS - 重要行为报告
- STAGE4_SETOFF_REVIEW - 抵消权审查

**阶段五：重整阶段**
- STAGE5_REORGANIZATION_PLAN - 重整计划
- STAGE5_REORGANIZATION_EXECUTION - 重整执行

**阶段六：分配阶段**
- STAGE6_BANKRUPTCY_DIST_PLAN - 破产财产分配方案
- STAGE6_EMPLOYEE_SETTLEMENT - 职工安置方案
- STAGE6_PRIORITY_PAYMENT - 优先受偿
- STAGE6_PROPERTY_DISTR_EXEC - 财产分配执行
- STAGE6_DEPOSIT_MANAGEMENT - 提存管理
- STAGE6_BANKRUPTCY_TERMINATION - 破产程序终结

**阶段七：终结阶段**
- STAGE7_CANCELLATION_REG - 注销登记
- STAGE7_TERMINATION_LITIGATION - 终结诉讼仲裁
- STAGE7_ADDITIONAL_DISTRIBUTION - 追加分配
- STAGE7_ACCOUNT_SEAL_MANAGE - 账户印章管理
- STAGE7_DUTY_REPORT - 职务报告
- STAGE7_DOCUMENT_TRANSFER - 资料移交
- STAGE7_ARCHIVING_MANAGEMENT - 归档管理
- STAGE7_SEAL_DESTRUCTION - 印章销毁
- STAGE7_ACCOUNT_CLOSING - 账户销户
